Practice managing multiple files in your Git repo.

Include or Exclude Changes to Compose Commits

As you learned earlier, a commit is a snapshot of changes made in a project. As you work, you make more and more commits. Commits become the project history, telling the story of the project's development.

A commit can include related changes from a single file or from many files. It's up to you! Generally we want to make smaller, more frequent commits to keep our history organized and understandable. Doing that will take some practice, and it will get easier with experience.

Let's practice managing changes in multiple files.

Create a new file

Remember the command for creating a new file from the command line? Create a file named "demo.txt" in your repo.

touch demo.txt

Now open your code editor and put some content in that file. As the file extension (".txt") tells us, this is a plain text file, so you can put any text you want here.

Make changes to another file

Now delete the last line of "readme.txt", and save the file again in your text editor.

Go back to your terminal and run git status to see what's changed. You should see the following: 


    On branch main
    Changes not staged for commit:
      (use "git add <file>..." to update what will be committed)
      (use "git restore <file>..." to discard changes in working directory)
      modified:   readme.txt

    Untracked files:
      (use "git add <file>..." to include in what will be committed)
      demo.txt

    no changes added to commit (use "git add" and/or "git commit -a")

Notice how your status gives you helpful suggestions for relevant commands. It's a good idea to read and consider these, though you don't have to do any of these suggestions now (or ever).

For now, we will leave these changes unstaged while we take a closer look.

What's the diff?

We learned in an earlier lesson how to show the difference between our unstaged changes and the last commit. Like git status, this command doesn't change anything. It just reports back to you with some helpful information. Let's check it out now:

git diff

Read the output in your terminal.

Notice that Git is showing you the deleted content from the "readme.txt" file, but the diff doesn't show anything about "demo.txt".

That is because "demo.txt" is still "untracked" -- it's a new file that Git hasn't started tracking yet, so it can't tell you what's changed. It's all new, and there is no diff yet!

Stage some changes but don't commit yet

What if we wanted to commit changes in some files but not others? We want to compose commits thoughtfully with related changes, so sometimes it makes sense to exclude some work and save it for a later commit.

Let's stage the changes only from "demo.txt" and leave out the changes from "readme.txt". How would you do this?

Here's what you will see if you do this step correctly:


    On branch main
    Changes to be committed:
      (use "git restore --staged <file>..." to unstage)
      new file:   demo.txt

    Changes not staged for commit:
      (use "git add <file>..." to update what will be committed)
      (use "git restore <file>..." to discard changes in working directory)
      modified:   readme.txt

Give it a go -- and if you need a reminder, look at earlier steps for guidance!

Troubleshooting

Here are some possible reasons you haven't passed this challenge.

You didn't stage the new file

To stage a file, you need to use this command:

git add <FILENAME>

You staged readme.txt

We can unstage this file while leaving the changes as they are. To unstage a file, use this command:

git reset <FILENAME>

You didn't make changes in both files

Open up your code editor and make the changes as instructed above.

You staged the file but the challenge still fails.

It's possible that your filename is incorrect. Check for whitespace and letters that should not be capitalized. For example, "Demo.txt" with a capital "D" is not the same as "demo.txt".

You committed your changes.

This challenge is only looking for staged changes. So, if you staged your changes and then went ahead and committed, the changes have moved from being staged to being committed, and so git doesn't find any changes in the "staging area". If this happens to you, just repeat the step above to make changes in "demo.txt", then stage the changes in that file, and click "Verify" again.
