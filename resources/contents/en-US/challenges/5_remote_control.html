Repositories Here and There

Earlier you learned about Git repositories and learned how to turn a project directory on your computer into a Git repository. In order to share your work with others, you can use GitHub as a central repository for code that a team of developers work on together.

In order to share code, you need a copy of your Git repository in a place where others can access it. That's what GitHub provides: a place to keep a shared repo. With a repo on GitHub, you can push your changes to it and pull down changes from others.

The GitHub copy of the Git repository on your computer is called a remote repository; it is not on your computer, but on a server, "remote" and somewhere else. The Git respository on your computer is called your local respository. You'll need to manage both and keep them in sync.

Other developers can always get the latest from your project by pulling your changes down from the remote (and onto their computer). This is how everyone can work on a project together without needing access to your computer, where your local copy is stored.

Create a Remote Repository

You want to sync your local repository with the one on GitHub. We already have a local repo, so we'll need to create a GitHub repo.

üíÅ You don't always start with a local repo. Sometimes you may start with a remote repo and have to make a local copy. It works both ways!

First, we'll create a new remote repository on GitHub. It will be empty to start with.

  1. Go to github.com, log in, and click the "+" in the top right and then select "New repository".
  2. Give it a name that matches your local repository's name (hello-world), and a short description.
  3. Make it public. This means it'll be listed on your public profile.
  4. Don't initialize with a README because we already have a file, locally, named "readme.txt". This is a helper option from GitHub if you hadn't already made it.
  5. Leave ".gitignore" and "license" set to "none". We won't use them for this tutorial.
  6. Click "Create repository"!

README files, Licenses, and .gitignore

These are common files in open source projects. To make starting a new project easier, GitHub provides an option to create them automatically.

A README file explains what the project is, how to use it, and often times, how to contribute (though sometimes there is an extra file, "CONTRIBUTING.md", for those details). The .md extension means it's a Markdown file. Markdown is a simple way to format text that looks great on GitHub.

A .gitignore file contains a list of filenames that Git should not track; for instance, files with passwords!

A license file describes the permitted use of the contents of the repo. It's optional to include one, but open source software authors often include it to define the fair use of their work. You can learn more about licenses in GitHub's documentation.

Connect your Local to your Remote

You should be on the main page for your empty repository on GitHub. You should see a highlighted section labelled 'Quick Setup'. In that section, make sure the 'HTTPS' button is selected. Copy the URL shown; this is the address of your repository on GitHub's servers.

Back in your terminal, and inside of the "hello-world" folder that you initialized as a Git repository in the earlier challenge, you want to tell Git to record the address of the remote version on GitHub's servers. It's possible to have multiple remotes, so each remote requires a name. The primary remote is typically named origin.

To add a remote named "origin" to your repository, run this command and paste in the URL you just copied:

git remote add origin your-URL-goes-here

Your local repository now knows where your remote repository lives on GitHub's servers. Think of it as adding a name ("origin") and address (the URL of your repo) on speed dial. This lets your local repo communicate easily with the remote repo.

Push Work to your Remote

Next you want to push (send) everything you've done locally to your remote repository on GitHub. This is something you'll do often, so that your remote repository is up to date and matches the state of your local repository.

Git has a system for working on a project in an orderly way, allowing you to "branch" off and work on the project without interfering with what's already done. We'll learn more about that later.

For now you just need to know that work in Git is always done on a branch. Branches contain commits. The first branch is named 'main' by default. When you push (and later pull) from a project, you tell Git the branch name you want and the name of the remote that it lives on.

Your local Git repo already has a branch named main, and it has one commit on it that you made in an earlier challenge. We want that commit and the code it includes to be copied into the remote repo on GitHub.

To create a branch named "main" in our remote repo on GitHub, we "push" our branch to the remote like this:

git push origin main

üíÅ In this case, "origin" is the name of the remote, and "main" is the name of the branch. The arguments to the push command are always in this order: git push remote-name branch-name.

Now go to your remote repository on GitHub (in your browser) and refresh the page.

Wow! The work you did locally is now on GitHub! üéâ Congrats on your first public repository!
