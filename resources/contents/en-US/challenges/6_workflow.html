Practice adding, committing, and pushing your changes to GitHub.

Workflow

A workflow is a set of practices that developers working together agree upon and follow to ensure that changes to the codebase are properly tracked, reviewed, and integrated into the project.

Different engineering organizations or teams may have different workflows adapted to the specific kind of work they are doing. At Momentum, while you are learning how to write code, you'll follow a short and simple workflow. Later on, you'll learn a more complicated one.

The basic workflow you will follow for your projects looks like this:

  1. Clone a project repo from GitHub.
  2. Work on your code in your editor.
  3. In your local repository, add changes as you work.
  4. Commit the changes in your local repository.
  5. Push the changes to the project repo on GitHub.

Let's practice doing this some more so that it starts to become familiar.

Make changes in your project

Let's open up the project in your code editor again and make some changes!

In your terminal, make sure you are still in your project directory. Remember that to open your "hello-world" project in VS Code, you can run this command:

code .

Edit the text in the "readme.txt" file you created earlier. The specific text doesn't matter. Perhaps a haiku?

Git add, commit, push. Tracks of changes in the code, History written.

Don't forget to save your changes.

Check Status

Back in the terminal, you can manage the changes you have made and decide when to commit them. The status command will give you a report on what's changed. You should get in the habit of checking it anytime you are thinking about making a commit.

Make sure you're still within your 'hello-world' directory when you're running these commands.

Check the status of your repo:

git status

You should see something like this: 


    On branch main
    Changes not staged for commit:
      (use "git add <file>..." to update what will be committed)
      (use "git restore <file>..." to discard changes in working directory)
            modified: readme.txt
    
    no changes added to commit (use "git add" and/or "git commit -a")

Add Changes

You should still be within your 'hello-world' directory for this step.

Remember that the add command means mark these changes to be included in the next commit. Another term used to describe adding files before committing in Git is "staging." You "stage" files to be committed.

Remember how to add a file?

git add readme.txt

Run git status again after you have run this command to confirm that your changes have been added.

You should see something like this:


    On branch main
    Changes to be committed:
    (use "git restore --staged ..." to unstage)
          modified: readme.txt

The filename highlighted in green indicates that the changes in that file will be included the next time you make a commit.

Commit Changes

Once you have staged your changes, you can make a commit any time you are ready. Anything that has been added (aka staged) will be included in the next commit.

You should still be within your 'hello-world' directory for this step.

To commit the changes to the repo's history , run this command. The message should describe why you made the changes.

git commit -m "Include a haiku in praise of git"

After you have run the commit command, run git status again. You should see the following:


    On branch main
    nothing to commit, working tree clean

This message lets you know that there aren't any uncommitted changes anymore, which is what we expect after making a commit!

Push to your Remote

You don't have to push after every commit. You can decide when you want to push. When you push, all the commits that you have made locally since the last time you pushed will be included. It's a good idea to do this often, so that your remote repo is always up to date, but when is up to you.

When you're ready, push your latest changes to the main branch on your remote repo.

git push origin main

Go to your repo's main page on GitHub. Above the section with the file(s) you should see a link to commits, with a number and an icon that looks like a clock with an arrow around it. If you click that link you will see a page with a list of your commits, identical to the list of commits in your local repo.

Click the 'Verify' button to confirm that committed your most recent changes.
